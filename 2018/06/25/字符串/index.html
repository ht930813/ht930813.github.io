##Java中的字符串##
本质为常量字符数组private final char value[];
String对象一旦被创建就是固定不变的了，对String对象的任何改变都不影响到原对象，相关的任何change操作都会生成新的对象

###字符串常量池###
字符串分配与其它对象分配一样，需要消耗时间和空间，而且字符串是我们使用非常多的。故JVM为了提高性能和减少内存的开销，在实例化字符串的时候进行了优化：使用字符串常量池，每当我们创建字符串常量时，JVM会首先检查字符串常量池，如果该字符串已经存在常量池中，那么就直接返回常量池中的实例引用。如果字符串不存在常量池中，就会实例化该字符串并且将其放到常量池中。由于String字符串的不可变性我们可以十分肯定常量池中一定不存在两个相同的字符串。
Java中的常量池，分为两种形态：静态常量池与运行时常量池

  • 静态常量池：界.class文件中的常量池，class文件中的常量不仅仅包含字符串/数字字面量，还包含类、方法的信息。这些占用class文件绝大部分空间。
  • 运行时常量池:即JVM在完成类的装载操作之后，将class文件中的常量池载入到内存中，并保存在方法区。（即为方法区中的运行时常量池）
    对于String类型，方法区有一张固定长度的CONSTANT_String_info表用来存储文字字符串值。常量池中保存着很多String对象; 并且可以被共享使用，因此它提高了效率
    String c=new String("aaa"。);采用new关键字新建一个对象时，JVM首先去字符串常量查找有无此串。若有：则不在常量池创建此串，直接在堆中创建一个”aaa”对象，然后把此对象的地址返给引用c；若无：首先在字符串常量池中创建此串对象，然后再堆中创建此串对象，最后把堆中此串对象的地址返给引用c。
    字符串的引用若在编译时无法确定，则在堆中创建此对象，并把此对象的地址返给此引用

new String()不能再编译时期确定，new String() 创建的字符串指向的时堆内存的对象中，它们有自己的地址空间。s2因为有后半部分new String(”world”)所以也无法在编译期确定，所以也是一个新创建对象”helloworld”的引用。

JVM对String str=”abc”对象放在常量池中是在编译时做的，而String str3=str1+str2是在运行时刻才能知道的。str1 + str2首先以最左边的字符串为参数创建StringBuilder对象，然后依次对右边进行append操作，最后将StringBuilder的最后一步toString()方法还原一个新的String对象”abcdef”，因此堆中开辟一块空间存放此对象。

在程序编译时期，JVM会把字面字符串的”+”连接优化为连接后的值，即编译时期就确定字符串常量的值。(为了避免创建多余的字符串对象而做的优化)，而上述的str1+str2效率要比直接字面字符串相加低，故编译器不会对引用变量进行优化。

对于final修饰的变量，它在编译时被解析为常量值的一个本地拷贝存储到自己的常量池中或嵌入到它的字节码流中。所以此时的”a” + s1和”a” + “b”效果是一样的。

这里面虽然将s1用final修饰了，但是由于其赋值是通过方法调用返回的，那么它的值只能在运行期间确定，因此s0和s2指向的不是同一个对象
String不可变的深入剖析:private final char value[];仅靠private和final是根本没有办法保证String不可变，因为final修饰的只是value，即value不可变，而value本质只是一个指向对内存的地址，但是value所指向堆内存的对象的内容是可以改变的。因此最关键的是，还需要String内部的方法都不能对value数组进行直接修改，都要拷贝到新的char[]数组 或者 直接new 新的String对象

final只对引用的”值”(即内存地址)有效，它迫使引用只能指向初始指向的那个对象，改变它的指向会导致编译期错误。至于它所指向的对象的变化，final是不负责的。

###String.intern()###
当调用 intern方法时，如果池已经包含一个等于此String对象的字符串（用equals(oject)方法确定。对于任意两个字符串 s 和 t，当且仅当 s.equals(t) 为 true 时，s.intern() == t.intern() 才为 true），则返回池中的字符串。否则，将此String对象添加到池中，并返回此String对象的引用。

存在于.class文件中的常量池，在运行期间被jvm装载，并且可以扩充。String的intern()方法就是扩充常量池的一个方法；当一个String实例str调用intern()方法时，java查找常量池中是否有相同unicode的字符串常量，如果有，则返回其引用，如果没有，则在常量池中增加一个unicode等于str的字符串并返回它的引用。


String采用连接运算符”+”效率低下原因分析:每做一次 + 就产生个StringBuilder对象，最后将StringBuilder的最后一步toString()方法还原一个新的String对象。下次循环再到达时重新产生个StringBuilder对象，然后 append 字符串，最后将StringBuilder的最后一步toString()方法还原一个新的String对象,如此循环直至结束。如果我们直接采用 StringBuilder 对象进行 append 的话，我们可以节省 N - 1 次创建和销毁对象的时间。所以对于在循环中要进行字符串连接的应用，一般都是用StringBuffer或StringBulider对象来进行append操作。

String、StringBuffer、StringBuilder的区别

  • 可变与不可变：String是不可变字符串对象，StringBuilder和StringBuffer是可变字符串对象（其内部的字符数组长度可变）
  • 线程是否安全：String中的对象是不可变的，也就可以理解为常量，显然线程安全。StringBuffer 与 StringBuilder 中的方法和功能完全是等价的，只是StringBuffer 中的方法大都采用了synchronized 关键字进行修饰，因此是线程安全的，而 StringBuilder 没有这个修饰，可以被认为是非线程安全的。
  • 执行效率：StringBuilder > StringBuffer > String 当然这个是相对的。这三个类是各有利弊，应当根据不同的情况来进行选择使用：
    • 当字符串相加操作或者改动较少的情况下，建议使用 String str=”hello”这种形式；
    • 当字符串相加操作较多的情况下，建议使用StringBuilder，如果采用了多线程，则使用StringBuffer。

关于String str = new String(“abc”)创建了多少个对象:在类加载过程中，在运行时的常量池中创建了”abc”对象，而在代码运行期间在堆创建了一个”abc”对象。
字符串池的优缺点：避免了相同内容的字符串的创建，节省了内存，省去了创建相同字符串的时间，同时提升了性能；另一方面，字符串池的缺点就是牺牲了JVM在常量池中遍历对象所需要的时间，不过其时间成本相比而言比较低。
